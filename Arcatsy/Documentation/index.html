ARchive CATalogue SYstem

ArCatSy has been designed primarily to catalogue collections in an archive. But has been made such, that any type of information about practically anything can be organised and made accessible by means of ordered lists (or catalogues), by indices or registers and in online queries.

ArCatSy provides for:

  • Definition of the fields used in a collection, with the data entry order and indicators if programmable exits in Rexx are to be inserted in the processing

  • Definition of the layout in which the data of collections is shown on the screen, when searching, or is printed on paper. The markup language is html, so flexibility is guaranteed.

  • A data entry system, allowing for several different layouts for inserting the data. The data entry process is guided by a procedure written in the Rexx language, and the default Rexx data entry procedure is generated for you by the system. It only needs refinement to allow a smooth data entry, as nice as you want it. The procedure provides for a pure data entry with blank screens to start from and a correction mode on existing data that you select from a list, ordered according to your wishes.

  • A table lookup system, for use in the data entry, to prepare lists of common used texts.

  • Searching through the data, with 1 to 10 terms, in a & | or ^ (and, or, not) context. Possibility to limit the search in specific collections. Available search terms are based on extracting pieces of search terms from a text field by using regular expressions, as defined by Microsoft for the .Net programming environment. Most useful examples are included in the documentation of ArCatSy. You might define every field to be searched by on every word.

  • Indices or registers on the information of one or more collections, you choose which field(s) to use as source, and what regular expression to use to extract terms from that source.

  • Ordering results of a search on any number of fields in a collection and print the resulting ordered list.

  • Translation of the package in any language. All literal texts are stored in one text file.

  • Maintenance procedures, for import and export of data and definitions, and what else is needed to make the system function well.
